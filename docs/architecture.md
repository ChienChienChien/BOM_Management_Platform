# BOM 智慧運算與管理平台｜架構圖

## 整體架構

```mermaid
flowchart TB
    subgraph Owners[參數與業務資料]
        A["工業工程／品保：原料成分"]
        B["採購：價格與採購限制"]
        C["運籌：生產計畫"]
        D["煉鋼：現場排程與即時需求"]
    end

    E["參數管理與簽核：應用介面、流程自動化"]
    F["參數資料庫與版本紀錄"]
    G["BOM 運算協調層：資料整合、排程、即時觸發"]
    H["團隊開發的 BOM 核心模型"]
    I["結果與輸入快照：版本、參數、驗證紀錄"]

    subgraph Outputs[發布與應用]

    end

    A --> E
    B --> E
    C --> E

    G --> H
    H --> I
    I --> J
    I --> K
    J --> L
    K --> L
    I -. "異常通知與版本驗證" .-> E
```

## 各層職責

| 層級 | 主要職責 |
|---|---|
| 參數與業務資料 | 由權責單位提供原料成分、價格、採購限制、生產計畫及現場排程 |
| 參數管理與簽核 | 提供維護介面，執行申請、簽核、通知及正式資料更新 |
| 參數資料庫與版本紀錄 | 保存正式參數、異動前後內容、簽核狀態與歷史版本 |
| BOM 運算協調層 | 整合資料、檢查輸入、處理排程或即時觸發並呼叫核心模型 |
| BOM 核心模型 | 由團隊成員主責開發，根據輸入條件產出用料結果 |
| 結果與輸入快照 | 同步保存模型結果及本次運算使用的參數、計畫與排程 |
| 發布與應用 | 發布 BOM1／BOM2，並提供下游成本與庫存預測使用 |

## 參數異動流程

```mermaid
stateDiagram-v2
    state "提出異動" as Request
    state "權責簽核" as Approval
    state "更新正式參數" as Update
    state "執行 BOM" as Run
    state "驗證結果" as Validate
    state "發布版本" as Publish
    state "保存紀錄" as Record

    [*] --> Request
    Request --> Approval: 提交異動內容
    Approval --> Request: 退回修正
    Approval --> Update: 核准
    Update --> Run: 套用正式參數
    Run --> Validate: 產生試算結果
    Validate --> Run: 發現異常
    Validate --> Publish: 驗證通過
    Publish --> Record: 發布正式結果
    Record --> [*]
```

## BOM1 與 BOM2

| 項目 | BOM1 | BOM2 |
|---|---|---|
| 主要用途 | 採購與中期規劃 | 現場即時用料決策 |
| 更新方式 | 每週定期更新 | 由現場系統即時觸發 |
| 主要輸入 | 正式參數、生產計畫 | 正式參數、現場排程與即時需求 |
| 共同機制 | 參數版本、輸入快照、結果紀錄、驗證與異常通知 | 參數版本、輸入快照、結果紀錄、驗證與異常通知 |

## 追溯關係

```mermaid
flowchart LR
    A["BOM 結果版本"] --> B["本次運算紀錄"]
    B --> C["參數版本"]
    B --> D["生產計畫或排程"]
    B --> E["模型輸入與檢核結果"]
    C --> F["異動與簽核紀錄"]
```

透過以上關係，使用者可從異常結果反查本次運算、參數版本、計畫／排程及參數異動來源，避免只保留輸出檔案而無法還原當時條件。
